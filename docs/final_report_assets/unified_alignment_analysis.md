- **SFT** fits the reference distribution via supervised CE on `summary_en_chosen`.
- **DPO retune v2** aligns preferences from chosen/rejected pairs; in this project it can **match or slightly exceed** SFT on aggregate test metrics after conservative tuning.
- **GRPO-V1** optimizes a **weak** proxy (tags): good for demonstrating **misalignment** and hacking.
- **GRPO-V4** is a **stronger** proxy (structure + overlap with source) but still **not** full human judgment.

## 2. Why V1 is easier to hack

V1 = average of three booleans (tags present). Any completion containing `[point]`, `[reason]`, `[summary]` in any order of appearance in the string still gives **full tag credit** for `_tags_present` (substring check). Repeated blocks do not reduce V1. Therefore **reward hacking** is almost **by design** for Part III pedagogy.

## 3. Why V4 mitigates but does not solve hacking

V4 builds on order, numbered reasons (heuristic), length/repetition penalties, and overlap with `answer_en`. That blocks **some** degenerate policies but:

- Overlap can reward **copying** or **keyword stuffing**.
- Strict **single-template** constraints are **not** encoded in V4.
- **Strict format** in evaluation is stricter than the reward’s numbered-reason check in edge cases.

So V4 is **better aligned** than V1 with summary quality proxies, yet **reward hacking cases can remain** (see per-model hacking markdown).

## 4. Loose vs strict format — lesson for the whole project

**Loose** format (legacy) tracked whether tags existed and coarse order; **repeated legal templates** could still look “perfect.” **Strict** format enforces **exactly one** of each section and numbered reasons inside `[reason]`, matching the assignment’s intended single response. For any RL stage, **always report both**: optimization targets loose proxies; **strict** reveals silent regressions.

## 5. Three alignment paradigms (trade-offs)

1. **Supervised (SFT):** Stable, directly targets reference summaries; risk of **exposure bias** / limited exploration.
2. **Preference (DPO):** Can improve concision/format if pairs are clean; **bad rejects** or **too-strong β** can destroy quality (seen in original DPO before retune).
3. **Online RL (GRPO):** Flexible reward engineering; **high variance**, **reward hacking**, and need for **constraints** (length, format audits) and **sensible initialization** (here: **merged SFT** + small LoRA for GRPO).

## 6. Honesty clause

If GRPO **underperforms** SFT or DPO on ROUGE-L or strict on this split, that is still a **valid result**: the point of Part III is often to **show** how reward choice changes behavior and where metrics disagree. Do not hide worse numbers; explain them with **reward definition** and **hacking examples**.
