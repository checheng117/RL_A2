# GRPO qualitative analysis (Part III)

Numbers below come from **test split greedy evaluation** (`*_test_metrics.json`), same decode settings as SFT.

## 1. GRPO-V1 behavior

- **ROUGE-L** ≈ 0.4396 vs SFT ≈ 0.4408.
- **strict_format_rate** ≈ 0.988 vs SFT ≈ 0.988.
- **avg_reward (v1)** ≈ 1.0000 (training objective proxy on test completions).

V1 rewards **tag presence** only. Models can earn high reward with **repeated `[point]/[reason]/[summary]` blocks** or **shallow text**, which **loose format** may still score as passing — **strict format** is needed to expose that failure mode.

## 2. GRPO-V4 behavior

- **ROUGE-L** ≈ 0.4495.
- **strict_format_rate** ≈ 1.000.
- **avg_reward (v4)** ≈ 0.4992.

V4 adds ordering, numbered-reason heuristics, length/repetition shaping, and token overlap with `answer_en`. It **reduces some V1 hacks** but is **not identical** to strict evaluation and can still be gamed (e.g. keyword overlap without good summarization).

## 3. Reward hacking: V1 vs V4

- **V1** is intentionally **easy to hack** for the assignment: maximize tags with minimal semantic work.
- **V4** typically lowers reward on **tag-soup** and **extreme repetition**, but **cannot guarantee** human-quality summaries.

See concrete rows in:

- `outputs/report_assets/grpo_v1_reward_hacking_cases.md`
- `outputs/report_assets/grpo_v4_reward_hacking_cases.md`

## 4. Success vs failure (how to read generations)

- **Success-like**: single template, 2+ numbered reasons, concise `[summary]`, ROUGE-L not collapsed vs `summary_en_chosen`, **strict pass**.
- **Failure-like**: multiple full templates, empty reasons, or high reward with **strict fail** / low ROUGE-L.

## 5. Did strict metric help?

**Yes, as an external audit:** loose format stayed high for broken DPO runs earlier; strict format surfaced **duplicate legal blocks**. The same lens should be applied to GRPO outputs even though **RL optimized V1/V4 rewards, not strict**.
